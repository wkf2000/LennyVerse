# Neo4j Ingestion Design (P0)

Date: 2026-03-21  
Status: Approved for planning  
Scope: Implement Neo4j projection in ingestion pipeline using Option A

## 1) Goals and Non-Goals

### Goals

- Implement real Neo4j projection in the existing `project` stage of local ingestion.
- Keep Supabase Postgres as canonical source of truth; Neo4j remains a derived projection.
- Use stable IDs and idempotent upserts for rerun safety.
- Implement all target P0 relationships:
  - `(Document)-[:FEATURES_GUEST]->(Guest)`
  - `(Document)-[:HAS_TAG]->(Tag)`
  - `(Document)-[:MENTIONS_CONCEPT {weight, evidence_count}]->(Concept)`
  - `(Document)-[:USES_FRAMEWORK {weight, evidence_count}]->(Framework)`
  - `(Concept)-[:RELATED_TO {weight, method}]->(Concept)`
- Enforce fail-run semantics: projection failure causes non-zero exit.

### Non-goals

- No distributed orchestration, real-time ingestion, or CI ingestion automation.
- No change to core API fallback behavior that keeps search/learning usable without Neo4j.
- No expansion into P1 visualization API contracts in this task.

## 2) Architecture and Boundaries

### Chosen approach

Use **Option A**: add a dedicated projector module (`ingest/neo4j_projector.py`) and invoke it from `run_pipeline` when `project` is enabled.

### Module responsibilities

- `ingest/neo4j_projector.py`
  - Load Neo4j credentials from env and optional `.env`.
  - Create/verify uniqueness constraints on node IDs.
  - Upsert nodes and relationships in batches using Cypher `MERGE`.
  - Return projection stats (rows processed, per-entity counts, elapsed time).
- `ingest/pipeline.py`
  - Keep ownership of parse/chunk/embed/extract/load flow.
  - Build projection input payload from in-memory run artifacts.
  - Call projector in `project` stage and surface projection results in run payload.
- `ingest/cli.py`
  - Keep `run` and `backfill` shape.
  - Replace `rebuild-graph` marker behavior with real canonical rebuild execution.

### Source of projection data

- `ingest run`: project from in-memory payloads generated in the same run.
- `ingest rebuild-graph`: read canonical records from Supabase and reproject entire graph.

## 3) Data Mapping and Idempotency

## Stable IDs

Use existing stable IDs already generated by ingestion:

- `doc:{source_slug}`
- `guest:{slug(name)}`
- `tag:{slug(name)}`
- `concept:{slug(normalized_name)}`
- `framework:{slug(name)}`

## Node projection

- `Document {id, source_slug, source_type, title, published_at, updated_at}`
- `Guest {id, name}`
- `Tag {id, name}`
- `Concept {id, name, normalized_name, description}`
- `Framework {id, name, summary, confidence}`

Unknown or optional fields are omitted rather than set to unstable defaults.

## Relationship projection

- `FEATURES_GUEST` from `document_guests`.
- `HAS_TAG` from `document_tags`.
- `MENTIONS_CONCEPT` from `chunk_concepts` rolled up to document-level pairs:
  - `weight`: aggregated confidence score per `(document_id, concept_id)`
  - `evidence_count`: number of chunk evidences for the pair
- `USES_FRAMEWORK` from `chunk_frameworks` rolled up to document-level pairs:
  - `weight`: aggregated confidence score per `(document_id, framework_id)`
  - `evidence_count`: number of chunk evidences for the pair
- `RELATED_TO` between concepts derived from co-occurrence in the same document:
  - `weight`: normalized co-occurrence count
  - `method`: fixed `"cooccurrence_p0"`

## Idempotent write contract

- All nodes use `MERGE` by `id`.
- All relationships use `MERGE` by endpoint IDs + relationship type, then property `SET`.
- Constraints are created with idempotent Cypher (`IF NOT EXISTS`) for each label.
- Re-running the same input must be safe and converge to the same graph state.

## 4) Failure Semantics and Run State

### Fail-run behavior

- Any Neo4j connectivity, auth, constraint, or query error in `project` stage raises an exception.
- CLI exits non-zero for that run.
- Error details are included in run payload for diagnosis.

### Checkpoint ordering

- `checkpoint_state.json` is updated only after successful projection when `project` is enabled.
- This prevents marking documents as fully processed when graph projection failed.

### Artifact behavior

- Existing JSON artifacts (`documents.json`, `chunks.json`, `extractions.json`, `last_run.json`) are still written as applicable.
- `last_run.json` includes projection summary or projection error block.

## 5) CLI and Command Contracts

### `ingest run`

- If `project` stage is selected, run Neo4j projection from in-memory payloads.
- Include `projection_result` in JSON output on success.
- Exit non-zero on projection failure.

### `ingest backfill`

- Same projection behavior as `run`, scoped by source filter.

### `ingest rebuild-graph`

- Replace marker creation with real rebuild execution:
  1. Read canonical entities and joins from Supabase.
  2. Ensure Neo4j constraints.
  3. Reproject all nodes and relationships.
  4. Print rebuild summary counts and duration.
- Exit non-zero on any projection failure.

## 6) Implementation Plan (High Level)

1. Add `ingest/neo4j_projector.py` with:
   - env loading
   - driver/session management
   - constraint setup
   - batch upsert helpers for nodes and edges
2. Extend `run_pipeline`:
   - build projector inputs from existing payload structures
   - call projector in `project` stage
   - include projection result in `run_payload`
   - move checkpoint update after project success
3. Extend CLI `rebuild-graph`:
   - remove marker behavior
   - implement canonical fetch + projector invocation
4. Add tests for projector behavior and pipeline integration.

## 7) Testing Strategy

### Unit tests

- Concept/framework rollup into document-level weighted edges.
- Concept co-occurrence to `RELATED_TO` edge generation.
- Stable ID mapping and row-shape validation for projection batches.

### Integration tests (mocked external clients)

- Successful `project` stage returns projection stats and completes run.
- Projection failure causes non-zero behavior and preserves failure diagnostics.
- Checkpoint update only after successful projection.
- `rebuild-graph` executes canonical fetch + projection path (no marker file behavior).

### Manual verification

- Run local ingestion on sample corpus with Neo4j available.
- Validate constraints, node counts, and relationship counts in Neo4j browser.
- Re-run ingestion unchanged and verify idempotent counts/state.

## 8) Risks and Mitigations

- **Risk:** Neo4j write throughput bottlenecks on large backfills.  
  **Mitigation:** batch writes and keep relationship rollups in memory before upsert.
- **Risk:** Co-occurrence logic creates noisy `RELATED_TO` edges.  
  **Mitigation:** keep method tagged (`cooccurrence_p0`) and tune threshold/normalization later.
- **Risk:** Drift between canonical and graph projection over time.  
  **Mitigation:** keep rebuild path canonical-driven and run periodic rebuild/drift checks.

## 9) Acceptance Criteria

- `project` stage performs real Neo4j writes for all required nodes and relationships.
- Projection uses stable IDs and is idempotent on rerun.
- `rebuild-graph` performs full canonical rebuild (no marker placeholder).
- Projection failures fail the run with actionable error context.
- Tests cover success path, failure path, and checkpoint correctness.
